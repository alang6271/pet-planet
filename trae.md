## 一、项目概述

### 1.1 基本信息

| 项目 | 内容 |
|------|------|
| **项目名称** | 宠物星球 Pet-Planet |
| **产品定位** | 面向宠物主人的线上数字纪念平台 |
| **产品 Slogan** | 为逝去的毛孩子，点亮一颗永恒的星 |
| **目标用户** | 失去爱宠的宠物主人 |

### 1.2 背景描述

当一只宠物离开，主人的世界会空出一块。他们需要一个空间来寄托哀思、存放回忆——但市面上缺少专门为宠物打造的线上纪念平台。

宠物星球致力于为每一位失去爱宠的主人，提供专属的虚拟星球空间。在这里，每一只毛孩子都拥有一颗独一无二的星球，主人可以在星球上存放照片、记录故事、点亮蜡烛，完成一场跨越时空的告别与纪念。

### 1.3 核心价值

- **情感寄托**：为悲伤的情绪提供一个温暖的安放之处
- **永恒纪念**：将转瞬即逝的陪伴化为永久的数字记忆
- **仪式感**：通过点蜡烛等仪式化行为，辅助悲伤处理与心理愈合
- **独特性**：每颗星球都是独一无二的，象征每一段独特的宠物情缘

### 1.4 设计基调

| 维度 | 说明 |
|------|------|
| **整体风格** | 简约、温暖、治愈，参考 Apple 官网的留白美学 |
| **情感基调** | 温柔而不沉重，静谧而不冷漠 |
| **视觉主题** | 深邃星空 + 暖光点缀，营造浩瀚宇宙中的温暖归属感 |
| **主色调** | 深空蓝（背景）+ 暖金色/星光白（强调）+ 电光蓝（CTA按钮） |
| **按钮风格** | 胶囊形（capsule-shaped），圆润温和 |
| **字体** | 无衬线字体，简洁现代，中文使用思源黑体/苹方 |

## 二、页面设计

本项目至少包含以下三个核心页面，后续可根据需求扩展。

### 2.1 首页（产品介绍页）

**页面路径**：`/`

**设计目标**：向用户传达产品理念，激发情感共鸣，引导用户开始创建纪念星球。

**页面结构（从上到下）**：

1. **导航栏**
   - 左侧：Logo「宠物星球 Pet-Planet」
   - 右侧：导航链接（首页、星球、关于）、登录/注册按钮（胶囊形）

2. **Hero 主视觉区**
   - 全屏深空星空背景，缓慢飘动的星辰粒子
   - 居中大标题：「为逝去的毛孩子，点亮一颗永恒的星」
   - 副标题：一句温柔的产品描述
   - 主 CTA 按钮（电光蓝胶囊）：「创建纪念星球」
   - 次 CTA 按钮（描边胶囊）：「了解更多」

3. **产品理念区**
   - 三栏布局，每栏一个图标 + 标题 + 描述
   - 主题：永恒的陪伴 / 专属的星球 / 温暖的告别

4. **功能展示区**
   - 交替图文布局，展示三大核心功能
   - 每个功能配以示意动画/插图

5. **情感故事区（可选）**
   - 用户真实故事分享（后期填充）

6. **页脚**
   - 联系方式、隐私政策、版权信息

### 2.2 星球页（宇宙视图）

**页面路径**：`/planets`

**设计目标**：呈现浩瀚宇宙中漂浮的多颗星球，每颗星球代表一只被纪念的宠物，用户可点击进入对应宠物空间。

**页面结构**：

1. **顶部栏**
   - Logo + 返回首页
   - 用户头像/昵称
   - 「创建新星球」按钮（胶囊形）

2. **3D 宇宙场景（核心区域）**
   - 全屏 3D 渲染的深空场景
   - 多颗星球在宇宙中缓慢漂浮、自转
   - 每颗星球上方悬浮宠物名字标签
   - 星球外观各异（颜色、纹理、光环等可自定义）
   - 支持鼠标拖拽旋转视角、滚轮缩放
   - 鼠标悬停星球时高亮 + 显示宠物信息卡片（名字、日期、简短寄语）
   - 点击星球 → 进入该宠物空间

3. **侧边栏（可选）**
   - 星球列表（缩略图 + 名字）
   - 便于在星球较多时快速定位

4. **空状态**
   - 当用户尚未创建任何星球时，显示引导插画 + 文案
   - 「点亮第一颗星，开始永恒的陪伴」

### 2.3 宠物空间页

**页面路径**：`/pets/:id`

**设计目标**：进入某只宠物的专属空间，查看记忆档案、点亮蜡烛、寄托思念。

**页面结构**：

1. **空间头部**
   - 宠物名字（大字）+ 在世日期 + 彩虹桥日期
   - 宠物头像（圆形或星球造型）
   - 一句寄语/墓志铭

2. **星球可视化区**
   - 顶部展示该宠物的专属 3D 星球（小型）
   - 星球可旋转、可自定义外观

3. **记忆档案区（功能一）**
   - 时间线展示（参考 QQ 空间/朋友圈的纵向时间线设计）
   - 每条记忆卡片：照片 + 文字 + 日期 + 分类标签
   - 支持分类筛选：全部 / 幼年 / 成年 / 趣事 / 日常 / 告别
   - 「添加记忆」按钮 → 弹出上传面板（照片上传 + 文字输入 + 日期 + 分类）

4. **点蜡烛区（功能三）**
   - 专属区域展示蜡烛
   - 可点亮蜡烛，蜡烛带有柔和的火焰动画
   - 显示蜡烛数量 + 点亮者（可选匿名）
   - 「为TA点亮一支蜡烛」按钮

5. **留言墙（可选扩展）**
   - 亲友可以留言寄语

## 三、功能模块

### 功能一：宠物记忆档案 📷

**所在页面**：宠物空间页

**功能描述**：上传照片、文字，记录宠物的一生故事，打造专属的数字时光胶囊。

**核心价值**：系统化保存与爱宠的点点滴滴，让回忆不会随时间模糊。

**详细设计**：

| 子功能 | 说明 |
|--------|------|
| **照片上传** | 支持上传 JPG/PNG/HEIC 等格式，支持多图，自动压缩存储 |
| **文字记录** | 每条记忆可附加文字描述，无字数硬限制 |
| **时间线展示** | 纵向时间线，按日期排序，参考 QQ 空间的时光轴设计 |
| **分类管理** | 预设分类：幼年、成年、趣事、日常、告别；支持自定义分类 |
| **日期标记** | 每条记忆关联一个日期，可手动填写或选择 |
| **编辑/删除** | 支持对已发布记忆的编辑和删除 |

**交互流程**：
1. 用户点击「添加记忆」
2. 弹出上传面板（模态框）
3. 选择照片 → 输入文字 → 选择日期 → 选择分类
4. 点击「保存」→ 记忆出现在时间线对应位置
5. 时间线支持滚动浏览，支持按分类筛选

---

### 功能二：虚拟纪念星球 🪐

**所在页面**：星球页、宠物空间页

**功能描述**：为每只宠物创建一颗独特的虚拟 3D 星球，点击星球可进入宠物空间。

**核心价值**：每颗星球都是独一无二的，代表每一段独特的宠物情缘。

**详细设计**：

| 子功能 | 说明 |
|--------|------|
| **3D 星球展示** | 使用 WebGL 技术渲染真实感 3D 星球，支持自转 |
| **外观自定义** | 可选择星球颜色、纹理风格、光环、装饰元素 |
| **宇宙场景** | 多颗星球在深空背景中漂浮，星辰粒子点缀 |
| **交互操作** | 鼠标拖拽旋转视角、滚轮缩放、点击星球进入空间 |
| **悬停信息** | 鼠标悬停星球显示宠物信息卡片 |
| **星球命名** | 每颗星球以宠物名字命名 |

**星球自定义选项**：
- 🎨 颜色主题（预设色板：暖阳金、深海蓝、森林绿、樱花粉、星空紫等）
- 🌀 纹理风格（光滑、条纹、斑点、星云）
- 💫 光环（有/无，光环颜色）
- ✨ 装饰元素（星星、花瓣、光点）

---

### 功能三：点蜡烛 🕯️

**所在页面**：宠物空间页

**功能描述**：为宠物点亮一支蜡烛，寄托永恒的思念。

**核心价值**：仪式化的纪念行为有助于悲伤处理，点蜡烛是一种跨越文化的哀思表达。

**详细设计**：

| 子功能 | 说明 |
|--------|------|
| **点蜡烛动画** | 点击按钮后，蜡烛被点亮，火焰以柔和动画燃起 |
| **蜡烛展示** | 已点亮的蜡烛在专属区域展示，带有摇曳的火焰效果 |
| **蜡烛计数** | 显示累计点亮的蜡烛数量 |
| **点亮者信息** | 可显示点亮者昵称（支持匿名） |
| **留言附言** | 点蜡烛时可附带一句寄语（可选） |

**交互流程**：
1. 用户在宠物空间看到「为TA点亮一支蜡烛」按钮
2. 点击按钮 → 弹出小面板（可选填昵称、寄语）
3. 确认 → 蜡烛亮起动画 → 蜡烛出现在展示区
4. 蜡烛持续燃烧，火焰以 CSS/Canvas 动画摇曳

## 四、技术栈方案（待用户选择）

> ⚠️ **重要**：以下列出了 4 套完整的技术方案，每套方案均包含前端、后端、数据库、3D 技术的完整选型。  
> 请您阅读各方案的说明与优劣势后，选择其中一套。选定后我将以该方案为基础进行开发。

---

### 方案 A：React + Express 全栈方案

**方案概述**：采用 React 构建前端界面，Express 搭建后端 API，MongoDB 存储数据，Three.js 实现 3D 星球效果。这是最成熟、最灵活的组合方式。

| 层级 | 技术选型 | 说明 |
|------|----------|------|
| **前端框架** | React 18 + TypeScript | 主流前端框架，组件化开发 |
| **构建工具** | Vite | 极速开发服务器与构建 |
| **3D 引擎** | Three.js + React Three Fiber | 最强大的 WebGL 3D 库，React 生态集成好 |
| **后端框架** | Express + Node.js | 轻量灵活的 Node.js 后端框架 |
| **数据库** | MongoDB | 文档型数据库，适合存储灵活的记忆档案数据 |
| **文件存储** | 本地/阿里云 OSS | 存储用户上传的照片 |
| **部署** | 传统服务器 / Docker | 可部署在任意云服务器 |

**优势**：
- ✅ 生态最成熟，社区资源丰富，遇到问题容易找到解决方案
- ✅ 3D 效果最强，React Three Fiber 是目前最好的 React 3D 解决方案
- ✅ 架构清晰，前后端分离，便于后期扩展
- ✅ MongoDB 的文档结构非常适合存储宠物记忆档案（照片、文字、分类等灵活数据）
- ✅ 完全自主可控，数据存储在自己的服务器上

**劣势**：
- ❌ 需要自己搭建和维护较多基础设施（认证、文件上传、部署等）
- ❌ 首页等内容页面的 SEO（搜索引擎优化）较弱，不利于被搜索引擎收录
- ❌ 服务器需要持续维护，有一定运维成本

**适合场景**：希望完全掌控代码和数据，追求最强 3D 效果和最大灵活性的方案。

---

### 方案 B：Next.js 全栈方案

**方案概述**：采用 Next.js 框架（基于 React），前后端一体化开发，内置服务端渲染，部署简单。PostgreSQL 作为结构化数据库。

| 层级 | 技术选型 | 说明 |
|------|----------|------|
| **全栈框架** | Next.js 14 + TypeScript | React 全栈框架，前后端一体 |
| **3D 引擎** | Three.js + React Three Fiber | 同方案 A |
| **数据库** | PostgreSQL + Prisma ORM | 关系型数据库，数据结构严谨 |
| **文件存储** | Vercel Blob / 阿里云 OSS | 云端文件存储 |
| **部署** | Vercel 平台 | 一键部署，自动 HTTPS，全球 CDN |

**优势**：
- ✅ 部署极简，Vercel 平台一键部署，自动配置 HTTPS 和 CDN 加速
- ✅ 支持服务端渲染（SSR），首页等产品介绍页 SEO 好，容易被搜索引擎收录
- ✅ 前后端一体化，开发体验好，代码组织更紧凑
- ✅ PostgreSQL + Prisma 数据结构严谨，数据一致性有保障
- ✅ Vercel 平台提供免费额度，前期几乎零成本

**劣势**：
- ❌ 3D 场景与服务端渲染需要额外处理，有一定复杂度
- ❌ Vercel 是海外平台，国内访问速度可能受影响（可搭配国内 CDN 解决）
- ❌ Next.js 的学习曲线比纯 React 略陡
- ❌ 深度绑定 Vercel 生态，迁移成本较高

**适合场景**：希望快速上线、部署省心、且重视搜索引擎收录的方案。

---

### 方案 C：Vue 3 全栈方案

**方案概述**：采用 Vue 3 构建前端，Express 或 Nuxt.js 作为后端，MongoDB 存储数据。Vue 的中文社区非常活跃。

| 层级 | 技术选型 | 说明 |
|------|----------|------|
| **前端框架** | Vue 3 + TypeScript | 渐进式前端框架，中文文档完善 |
| **构建工具** | Vite | Vue 作者开发，与 Vue 配合最佳 |
| **3D 引擎** | Three.js + TresJS | Vue 生态的 Three.js 封装 |
| **后端框架** | Express + Node.js | 同方案 A |
| **数据库** | MongoDB | 同方案 A |
| **部署** | 传统服务器 / Docker | 同方案 A |

**优势**：
- ✅ Vue 中文社区非常活跃，中文文档和教程丰富
- ✅ Vue 的模板语法对初学者更友好
- ✅ Vite 由 Vue 作者开发，与 Vue 配合最默契
- ✅ 完全自主可控，部署灵活

**劣势**：
- ❌ Vue 的 3D 生态不如 React 丰富，TresJS 不如 React Three Fiber 成熟
- ❌ 3D 星球功能可能需要更多手写 Three.js 原生代码
- ❌ 整体生态规模小于 React，部分第三方库可能没有 Vue 版本
- ❌ 同样存在 SEO 较弱和运维成本的问题

**适合场景**：偏好 Vue 生态、注重中文社区支持的方案。

---

### 方案 D：BaaS 快速开发方案

**方案概述**：前端使用 React，后端使用 Supabase（开源后端即服务平台），无需自己编写后端代码，开发速度最快。

| 层级 | 技术选型 | 说明 |
|------|----------|------|
| **前端框架** | React 18 + TypeScript | 同方案 A |
| **构建工具** | Vite | 同方案 A |
| **3D 引擎** | Three.js + React Three Fiber | 同方案 A |
| **后端服务** | Supabase | 开源 BaaS，提供数据库、认证、存储、API |
| **数据库** | Supabase PostgreSQL | 自动提供 RESTful API |
| **文件存储** | Supabase Storage | 内置文件存储服务 |
| **部署** | Vercel / Netlify | 前端托管平台 |

**优势**：
- ✅ 开发速度最快，后端无需写代码，Supabase 自动生成 API
- ✅ 自带用户认证系统（注册/登录），无需自己实现
- ✅ 自带文件存储服务，照片上传开箱即用
- ✅ 自带数据库管理界面，可视化管理数据
- ✅ 前期免费额度充足，适合快速验证产品

**劣势**：
- ❌ 深度依赖 Supabase 平台，后期迁移成本较高
- ❌ 复杂业务逻辑实现受限，不如自建后端灵活
- ❌ Supabase 是海外服务，国内访问速度可能不稳定
- ❌ 免费额度有上限，用户量增长后需付费

**适合场景**：希望用最快速度做出可用产品、验证想法的方案。

## 五、项目协作规则

### 5.1 ⚠️ 核心规则：技术决策由用户最终决定

> **在整个项目开发过程中，用户不提供具体的技术实现要求。所有涉及技术的决策，必须由我（AI 助手）列出多个可选方案，并详细解释每个方案的优劣势，由用户阅读后选择决定。**

**具体执行方式**：

1. **遇到技术选型时**：我主动列出 2-4 个可选方案，每个方案包含：
   - 方案名称与简介
   - 具体技术组成
   - 优势（用 ✅ 标注）
   - 劣势（用 ❌ 标注）
   - 适合场景说明

2. **遇到技术实现方式选择时**：同样列出多个方案并解释，由用户决定。

3. **遇到技术问题/bug 时**：列出可能的解决方向，解释后由用户选择处理方式。

4. **用户选择后**：严格按照用户选定的方案执行，不再擅自更改技术选型。

5. **此规则贯穿整个项目周期**，包括但不限于：
   - 框架/库的版本选择
   - 第三方服务的选择（存储、CDN、认证等）
   - 部署方案的选择
   - 架构设计的选择
   - 性能优化方案的选择
   - 任何涉及技术实现细节的决策

### 5.2 沟通语言

- 所有沟通使用中文
- 代码注释使用中文
- 技术术语在首次出现时附带中文解释

### 5.3 设计风格

- 参考 Apple 官网美学：简约、留白、胶囊形按钮
- CTA 按钮使用电光蓝色调
- 布局：左侧导航分类，右侧内容/工具列表（工具类页面适用）
- 渐进式开发：从一个功能开始，逐步扩展

### 5.4 开发方式

- 增量开发，先完成核心功能再扩展
- 每个阶段完成后由用户确认，再进入下一阶段
- 代码保持简洁，不过度设计

## 七、已确认的技术决策

### 7.1 数据存储方案（已确认 ✅）

**选定方案**：方案 B —— SQLite（结构化数据） + 本地文件系统（照片）

**存储分工**：

| 数据类型 | 存储位置 | 说明 |
|----------|----------|------|
| **宠物文字信息** | SQLite 数据库 | 名字、品种、生日、离世日期、寄语、星球配置等 |
| **记忆档案文字** | SQLite 数据库 | 每条记忆的文字描述、日期、分类等 |
| **蜡烛记录** | SQLite 数据库 | 点亮者、寄语、时间等 |
| **宠物照片** | 本地文件系统 | 存放在服务器 `uploads/` 目录，按宠物 ID 分子目录 |
| **记忆照片** | 本地文件系统 | 同上，数据库中只存文件路径/URL |

**数据库表结构（初步设计）**：

```
pets（宠物表）
├── id              主键
├── name            宠物名字
├── species         品种
├── birth_date      生日
├── death_date      离世日期（彩虹桥日期）
├── avatar_path     头像文件路径
├── epitaph         寄语/墓志铭
├── planet_config   星球配置（颜色、纹理、光环等，JSON 格式）
├── created_at      创建时间
└── updated_at      更新时间

memories（记忆档案表）
├── id              主键
├── pet_id          关联宠物 ID
├── content         文字内容
├── image_paths     照片路径（JSON 数组，支持多图）
├── memory_date     记忆发生的日期
├── category        分类（幼年/成年/趣事/日常/告别）
└── created_at      创建时间

candles（蜡烛表）
├── id              主键
├── pet_id          关联宠物 ID
├── lighter_name    点亮者昵称（可为空=匿名）
├── message         寄语（可选）
└── created_at      点亮时间
```

**照片存储规则**：
- 存放路径：`uploads/pets/{pet_id}/{timestamp}_{filename}.ext`
- 上传时自动压缩（控制单张图片体积）
- 数据库中只存储相对路径，前端通过 `/uploads/...` 访问

## 九、已确认的技术栈（已确认 ✅）

### 9.1 选定方案：方案 A —— React + Express + SQLite

**选定理由**：用户需求为「最简单、最易实现的 demo」，且已选定 SQLite + 本地文件存储方案。方案 A 与该存储方案完美契合，不依赖海外服务，3D 效果最省力，中文资料最丰富，本地开发零成本。

### 9.2 完整技术栈

| 层级 | 技术选型 | 版本/说明 |
|------|----------|-----------|
| **前端框架** | React 18 + TypeScript | 组件化开发，类型安全 |
| **构建工具** | Vite | 极速热更新，开发体验好 |
| **3D 引擎** | Three.js + React Three Fiber + Drei | 最成熟的 React 3D 方案 |
| **前端路由** | React Router v6 | 单页应用路由 |
| **HTTP 请求** | Axios | 前后端数据交互 |
| **后端框架** | Express + Node.js | 轻量灵活的 API 服务 |
| **数据库** | SQLite（better-sqlite3 驱动） | 文件型数据库，零配置 |
| **文件上传** | Multer | Express 文件上传中间件 |
| **照片存储** | 本地文件系统 `uploads/` 目录 | 按宠物 ID 分子目录 |
| **开发语言** | TypeScript（前后端统一） | 类型安全 |

### 9.4 产品决策汇总

| 决策项 | 选择 | 说明 |
|--------|------|------|
| **技术栈** | React + Express + SQLite | 方案 A，demo 最简方案 |
| **数据存储** | SQLite + 本地文件系统 | 文字存数据库，照片存文件夹 |
| **用户系统** | 暂不做登录 | 用简单密码保护管理权限，后期扩展 |
| **访问权限** | 公开可访问 | 任何人可查看宠物空间、点蜡烛 |
| **多宠物** | 支持多个 | 一个用户可创建多个宠物星球 |

### 9.5 管理权限说明（无登录系统的替代方案）

由于暂不做用户登录系统，对「创建/编辑/删除」等管理操作，采用**简单密码保护**：

- 创建一个全局管理密码（可在后端配置文件中设置）
- 执行管理操作（创建星球、添加/编辑/删除记忆）时需输入该密码
- 查看宠物空间、点蜡烛等浏览行为无需密码
- 后期扩展为正式用户系统时，可平滑迁移
